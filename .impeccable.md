## Design Context

### Users
Open-source community of self-hosters installing this on their own infrastructure to monitor and control Hetzner Cloud servers. They land here from a GitHub README or a friend's recommendation — no hand-holding. The dashboard is usually the third tab on the right, alongside a terminal and a Grafana window.

Jobs they bring:
- "What did the rescaler do in the last 24h?"
- "Is server X healthy and running the right size?"
- "Create a window / manually trigger a rescale / fix a key"
- "Onboard a new project"

They expect the dashboard to act like software, not marketing — keyboard-driven where reasonable, dense where possible, no empty-state illustrations explaining what a server is.

### Brand Personality
**Methodical, weighty, deliberate.** The UI should feel like a precision instrument — solid surfaces, considered motion, no filler. Reference spectrum: Hetzner Robot control panel modernized on the dark side, meets Linear's restraint. Explicitly NOT SaaS-y, NOT playful, NOT bouncy.

### Aesthetic Direction

**Theme** — Dark-first. Off-black tinted backgrounds in cool slate (not true black, no #000). Off-white tinted text (cool, not warm). Light mode is a first-class citizen but the dashboard is primarily read in dark.

**Typography** — Pair a distinctive industrial display face with a workhorse body. Display reserved for H1/page titles and big metric numerals; body for everything else. Module scale fixed rem on product UI (no fluid type on dashboards); clamp() only on the landing/login/marketing surfaces. Line-length capped at ~70ch body, tighter metric for tabular data.

**Color** — Cool tinted neutrals dominate (≈60%), a single restrained secondary channel for chart axes and labels (≈30%), a single warm amber accent reserved for "this just changed / this needs your attention" (≈10%). Never neon, never gradient text, never cyan-on-dark defaults, never purple-to-blue gradients. Accent only appears when semantic state changes.

**Surfaces & borders** — Hairline 1px borders separate regions; no drop shadows on the dashboard itself. Corner radii lean tight (2–6px), never the rounded-everything 12–16px blob. Cards are flat panels with a single border, not floating layers. Side-stripe borders (>1px colored) are forbidden — they always read as a 2024 admin template.

**Motion** — One orchestrated entrance reveal per page load (staggered, fast). State changes animate (hover, focus, value change) — they don't bounce, they ease-out-quart and stop. Layout transitions use transform/opacity only. `prefers-reduced-motion` honored everywhere.

**Iconography** — Outline-only, 1.25–1.5px stroke, neutral color inheriting from text. No filled illustrations, no decorative icon + heading pairings.

**Data density** — Operators want information. Show the metric, the timestamp, the units; do not pad them with explanatory copy. Where a chart would be purely decorative rather than load-bearing, leave it out.

### Design Principles
1. **Earned hierarchy** — every element exists because a job requires it. If you can't say which job, remove it.
2. **Type does the work** — size, weight, and spacing carry hierarchy; color and decoration stay out of the way.
3. **Surface, don't float** — borders separate; shadows elevate (sparingly). The dashboard stays grounded.
4. **State before decoration** — the accent color announces "this just changed." It never decorates.
5. **Operator-first density** — when in doubt, show the data denser. A hobby project can be cozy; this is a tool that should be skim-friendly at arm's length.
