## Design Context

### Users
Open-source community of self-hosters installing this on their own infrastructure to monitor and control Hetzner Cloud servers. They land here from a GitHub README or a friend's recommendation — no hand-holding. The dashboard is usually the third tab on the right, alongside a terminal and a Grafana window.

Jobs they bring:
- "What did the rescaler do in the last 24h?"
- "Is server X healthy and running the right size?"
- "Create a window / manually trigger a rescale / fix a key"
- "Onboard a new project"

They expect the dashboard to act like software, not marketing — keyboard-driven where reasonable, dense where possible, no empty-state illustrations explaining what a server is.

### Brand Personality
**Methodical, weighty, deliberate.** The UI should feel like a precision instrument — solid surfaces, considered motion, no filler. Reference spectrum: Hetzner Robot control panel modernized on the dark side, meets Linear's restraint. Explicitly NOT SaaS-y, NOT playful, NOT bouncy.

### Aesthetic Direction

**Theme** — Dark-first. Off-black tinted backgrounds in cool slate (not true black, no #000). Off-white tinted text (cool, not warm). Light mode is a first-class citizen but the dashboard is primarily read in dark.

**Typography** — Pair a distinctive industrial display face with a workhorse body. Display reserved for H1/page titles and big metric numerals; body for everything else. Module scale fixed rem on product UI (no fluid type on dashboards); clamp() only on the landing/login/marketing surfaces. Line-length capped at ~70ch body, tighter metric for tabular data.

**Color** — Cool tinted neutrals dominate (≈60%), a single restrained secondary channel for chart axes and labels (≈30%), a single warm amber accent reserved for "this just changed / this needs your attention" (≈10%). Never neon, never gradient text, never cyan-on-dark defaults, never purple-to-blue gradients. Accent only appears when semantic state changes.

**Surfaces & borders** — Hairline 1px borders separate regions; no drop shadows on the dashboard itself. Corner radii lean tight (2–6px) — utilitarian first, never the rounded-everything 12–16px blob. "Tech" but not "overwhelmingly digital": chrome stays restrained — no glow, no neon trim, no arcade-style accents. Cards are flat panels with a single border, not floating layers. Side-stripe borders (>1px colored) are forbidden — they always read as a 2024 admin template.

**Layout & scope** — Every page answers one question. Decorative chrome stays off the canvas: no hero sections on product pages, no large illustrations above headings, no marketing-style intro paragraphs the user already knows. Layout is compact and data-dense, never cramped — purposeful whitespace only. Each page is scoped to a single job (overview, status, projects, …) and the layout reads top-to-bottom in that scope.

**Navigation** — The sidebar is the only primary navigation. No top tab bars, no breadcrumb chains, no horizontal menus. The sidebar carries the entire information architecture: collapsible groups, a small unselected/selected indicator on the active item, no decorative icons above section labels. Section labels in muted text, page labels in active text only when the user is on that page.

**Devices** — Desktop is the primary canvas (wide tables, multi-column dashboards, dense charts). Mobile must be **fully usable**, not a degraded version: the sidebar collapses into a drawer, the table becomes a list, the chart degrades to a sparkline or summary number — never hidden. Targets hit the 40px minimum so a thumb can hit them.

**Motion** — One orchestrated entrance reveal per page load (staggered, fast). State changes animate (hover, focus, value change) — they don't bounce, they ease-out-quart and stop. Layout transitions use transform/opacity only. `prefers-reduced-motion` honored everywhere.

**Iconography** — Outline-only, 1.25–1.5px stroke, neutral color inheriting from text. No filled illustrations, no decorative icon + heading pairings.

**Data density** — Operators want information. Show the metric, the timestamp, the units; do not pad them with explanatory copy. Where a chart would be purely decorative rather than load-bearing, leave it out.

### Design Principles
1. **Earned hierarchy** — every element exists because a job requires it. If you can't say which job, remove it.
2. **Type does the work** — size, weight, and spacing carry hierarchy; color and decoration stay out of the way.
3. **Surface, don't float** — borders separate; shadows elevate (sparingly). The dashboard stays grounded.
4. **State before decoration** — the accent color announces "this just changed." It never decorates.
5. **Operator-first density** — when in doubt, show the data denser. A hobby project can be cozy; this is a tool that should be skim-friendly at arm's length.
6. **Scope per page** — one page, one job. Resist the urge to make any page a tour.
7. **Mobile is a tier, not a fallback** — every screen must do its job on a phone, just at a smaller column count.

### Tech Stack & UI Library

- **Base**: SvelteKit 2 + Svelte 5 + Tailwind CSS v4 + bun:sqlite + Better Auth.
- **State / data**: `$state` runes, derived stores, layout server loads for server-controlled flags.
- **Icons**: lucide-svelte or similar outline set (1.25–1.5px stroke, no filled icons).
- **Charts**: ApexCharts (already present). Keep chart chrome minimal — no default rounded corner radius on areas, axis labels in secondary text, gridlines tinted to the border color.
- **Component library**: see [§ UI library choice](#ui-library-choice) below.

#### UI library choice

**Recommended: shadcn-svelte** (built on **bits-ui**, the Svelte port of Radix Primitives — i.e. Radix UI for Svelte), layered with bespoke primitives copied into the repo so we own every component's HTML/CSS.

Why:

- **Fits the brief by construction**. shadcn-svelte ships components without imposed styling. You copy them in and tune the Tailwind classes to match the .impeccable.md tokens (2–6px radii, hairline borders, no shadows, accent for state-only). Flowbite's defaults fight this brief; Preline's defaults fight this brief.
- **Bits-ui = Radix for Svelte**. Same accessibility primitives (focus traps, keyboard navigation, ARIA roles for dialogs/popovers/menus) that the user explicitly asked about. Radix Primitives are React-only — bits-ui is the equivalent for Svelte, same authors where they overlap.
- **No component library tax**. Zero runtime cost, no upgrade pressure, no breaking-change deprecations. We own the components in `src/lib/components/ui/`. A change to a button isn't a PR to a vendored package — it's a PR to our own tree.
- **Tailwind v4 ready**. shadcn-svelte added v4 support in late 2024. We're already on v4.
- **Composable**. Dialog, dropdown, popover, tooltip, tabs, accordion, sheet (drawer for mobile sidebar), select, switch, slider — every primitive an operator-facing app needs, accessible by default.
- **License**. MIT, matches the project's licensing intent.

What it does NOT give us: pre-styled "dashboard widgets" (no KPI card, no chart card, no project row). Those are bespoke to this app anyway — they'll live in `src/lib/components/` next to the existing `ServerCard`, `KpiCard`, `RescalingActivityChart`, `CostBreakdownChart`. The result: bespoke cards on top of well-built accessible primitives.

Why not the alternatives:

- **Radix UI (React)** — wrong runtime; this is Svelte. The closest Svelte analogue is bits-ui, and shadcn-svelte wraps it.
- **Preline** — the design language skews SaaS-marketing (gradient hero variants, drop-shadowy cards, default-10px+ radii). Every overridden component becomes an audit. Wrong fit for "methodical, weighty, utilitarian".
- **Flowbite v2 (current)** — flows against the brief by default. We'd be in a constant override loop.
- **Headless Tailwind plugins (headlessui-svelte)** — fewer primitives, no copy-paste composition model, less momentum in the SvelteKit community.
- **Build everything by hand** — viable but throws away work bits-ui already maintains (focus management, roving-tabindex for menu items, escape-to-close on dialogs).
